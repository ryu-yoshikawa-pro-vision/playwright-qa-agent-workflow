# Mini SaaS サンプル

このサンプルは、架空の Web アプリケーションである Mini SaaS を対象に、ログイン機能の QA ワークフローを一通り示すものです。

このサンプルは実行対象ではありません。AI エージェントに作業の流れを理解させるための読み物です。

## 対象アプリケーション

Mini SaaS は、ログイン後にダッシュボードを表示する架空の SaaS です。

主な画面は次の2つです。

- ログイン画面
- ダッシュボード画面

このサンプルでは、ログイン機能だけを対象にします。

## サンプルで示す流れ

1. 対象プロジェクトプロファイルで、接続先、認証、テスト実行、データ制約を明確にする。
2. サービス探索で、画面、機能候補、証跡を整理する。
3. 仕様カタログで、再利用できる画面仕様、フロー、ルールを整理する。
4. `login.plan.md` で、ログイン機能のテスト計画を作る。
5. `login.test-design.md` で、テスト技法を使ってテストケースに落とす。
6. `login.validation.md` で、生成に進めるかを判定する。
7. `login.coverage.md` で、設計 ID と実装済みテストの対応を整理する。
8. `generated-tests/login.spec.ts.md` で、生成される Playwright Test の形を示す。

## このサンプルで扱わないこと

- 実在する Web アプリケーションへのアクセス
- 実際のログイン認証
- 実行可能な Playwright Test の配置
- DB 初期化や本物のテストデータ作成
- CI 上での E2E 実行

## 実案件に移すときの考え方

実案件では、このサンプルをそのままコピーするのではなく、対象アプリケーションの実際の仕様、画面、ロール、データ制約、テスト実行コマンドに置き換えます。

特に、対象プロジェクトプロファイル、テスト計画、テスト設計、validation、coverage 台帳は、実際に確認した証跡に基づいて更新してください。
