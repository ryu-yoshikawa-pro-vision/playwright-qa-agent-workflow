# 作業前チェックリスト

このページでは、`playwright-qa-agent-workflow` を使う前に確認することを説明します。

ここを曖昧にしたまま始めると、AI エージェントは途中で止まるか、推測で誤った成果物を作る可能性があります。

## 1. 作業の目的を決める

最初に、今回の作業がどれに当たるかを決めます。

| やりたいこと                                      | 使う流れ                  |
| ------------------------------------------------- | ------------------------- |
| 対象アプリ全体を把握したい                        | サービス全体探索          |
| どの機能をテスト化するか洗い出したい              | サービス全体探索          |
| 1つの画面や機能のテスト計画を作りたい             | 機能単位ワークフロー      |
| 既にあるテスト設計から Playwright Test を作りたい | validation 後に generator |
| 失敗した Playwright Test を直したい               | healer                    |

迷った場合は、いきなりテスト生成に進まず、サービス全体探索または機能単位の計画から始めます。

## 2. 対象アプリにアクセスできるか確認する

次の情報を確認してください。

| 確認項目         | 例                           | 不足している場合                     |
| ---------------- | ---------------------------- | ------------------------------------ |
| 対象アプリの URL | `https://stg.example.com`    | 探索を開始できません                 |
| 使用する環境     | dev、stg、検証環境           | 誤って本番を操作するリスクがあります |
| ログイン方法     | ID / パスワード、SSO、OAuth  | ログイン後画面を探索できません       |
| 使用するロール   | 一般ユーザー、管理者、閲覧者 | 権限差分を確認できません             |
| 操作してよい範囲 | 参照のみ、作成可、削除不可   | 危険な操作を避けられません           |

本番環境を使う場合は、作成、更新、削除、外部通知、メール送信、決済、招待などの操作可否を必ず明確にしてください。

## 3. 対象プロジェクトの情報を確認する

Playwright Test を生成または実行する可能性がある場合、対象プロジェクト側の情報が必要です。

| 確認項目             | 例                 | 説明                                        |
| -------------------- | ------------------ | ------------------------------------------- |
| 対象リポジトリ       | `owner/app-repo`   | テスト対象のコードがあるリポジトリ          |
| ブランチ             | `develop`          | どの状態に対して作業するか                  |
| ローカルパス         | `../app-repo`      | AI エージェントが読む対象プロジェクトの場所 |
| セットアップコマンド | `npm install`      | 初回準備に使うコマンド                      |
| 起動コマンド         | `npm run dev`      | ローカルでアプリを起動するコマンド          |
| テスト実行コマンド   | `npm run test:e2e` | 生成後の Playwright Test 実行に使うコマンド |
| 生成先               | `tests/e2e/`       | 生成したテストを置く場所                    |

このリポジトリは、対象アプリの起動方法やテスト実行コマンドを勝手に決めません。

対象プロジェクトに README、package scripts、CI 設定、既存テストがある場合は、そこから確認します。

## 4. データ操作ルールを確認する

AI エージェントは、テストのために画面を操作します。作成、更新、削除を伴う機能では、データ操作ルールが必要です。

最低限、次を決めてください。

| 確認項目           | 例                                   |
| ------------------ | ------------------------------------ |
| 作成してよいデータ | `AI_TEST_` で始まるデータのみ        |
| 更新してよいデータ | テスト用に作成したデータのみ         |
| 削除してよいデータ | 原則削除不可。必要時は承認後のみ     |
| 外部通知           | Slack、メール、Webhook は送信しない  |
| テスト後の片付け   | テストで作ったデータは最後に削除する |
| 共有環境の注意     | 他の利用者のデータには触れない       |

データ操作ルールが不明な場合、AI エージェントには安全側で `BLOCKED` と判断させます。

## 5. Playwright CLI が使えるか確認する

このリポジトリでは、画面探索や証跡取得に Playwright CLI を使います。

まず、次を実行します。

```bash
playwright-cli --help
```

実行できる場合、画面探索、snapshot、screenshot、trace などに進めます。

実行できない場合、次の作業は `BLOCKED` です。

- ブラウザで画面を開く
- 画面を操作する
- screenshot を撮る
- snapshot を取得する
- trace を取得する
- live UI を使って確認する

Playwright CLI が使えない状態で、AI エージェントに「確認したことにして」進めさせてはいけません。

## 6. このリポジトリの初期確認を行う

ローカルにリポジトリを用意したら、依存関係をインストールします。

```bash
npm install
```

次に、軽量チェックを実行します。

```bash
npm run check:evals
```

リポジトリ全体の lint、format、typecheck まで確認する場合は、次を実行します。

```bash
npm run check
```

ただし、これらのチェックは成果物構造や validation 整合性の確認です。

実際の対象アプリが正しく探索されたことや、生成したテストが意味的に正しいことまでは保証しません。

## 7. AI エージェントに渡す情報を整理する

AI エージェントへ依頼するときは、共通ルールではなく、今回の作業でしか分からない情報をまとめて渡します。

```text
対象は <対象アプリ名> です。
対象 URL は <URL> です。
使用する環境は <環境名> です。
使用するロールは <ロール名> です。
データ操作ルールは <ルール> です。
今回の目的は <サービス全体探索 / 機能単位の計画 / テスト設計 / 生成 / 修復> です。
参照すべき Target Project Profile は <path> です。
今回だけの注意点は <注意点> です。
```

AI エージェントは通常、`AGENTS.md` と該当スキルを読む前提で動きます。

そのため、利用者向けの依頼文では、共通指示を繰り返すよりも、対象 URL、ロール、データ操作ルール、feature slug、作業目的などの具体情報を明確にしてください。

この情報が不足している場合は、先に `02-target-project-profile.md` に従って対象プロジェクト情報を整理してください。

## 8. 作業開始前の最終確認

次にすべて答えられる状態で始めます。

- 対象アプリの URL は分かっているか
- 操作する環境は分かっているか
- 使用するアカウントやロールは分かっているか
- やってよい操作、やってはいけない操作は分かっているか
- 対象プロジェクトのテスト実行コマンドは分かっているか
- Playwright CLI は使えるか
- 成果物をどこに残すか分かっているか
- 今回使うスキルは分かっているか

1つでも作業に影響する重要項目が不明な場合は、推測で進めず、対象の作業を `BLOCKED` として扱います。
