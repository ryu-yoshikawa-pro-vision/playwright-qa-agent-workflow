# Agent Compatibility Guide

This repository can be used with agents in three levels of support.

## Level 1: Common agent layer

Use this layer for Codex and for future cross-agent workflow improvements:

| Purpose | Files |
|---|---|
| Repo instructions | `AGENTS.md` |
| MCP server config | `.codex/config.toml` |
| Reusable workflows | `.agents/skills/*/SKILL.md` |
| Workflow documentation | `docs/` |

Common skills:

- `playwright-service-mapper`
- `playwright-test-planner`
- `playwright-test-plan-validator`
- `playwright-test-generator`
- `playwright-test-healer`

Prefer maintaining this layer instead of editing tool-specific generated files.

## Level 2: Playwright-generated tool-specific support

Use these when available:

| Agent environment | Files |
|---|---|
| Claude Code | `.claude/agents/*.md` |
| GitHub Copilot / VS Code | `.github/agents/*.agent.md`, `.vscode/mcp.json` |
| OpenCode | `.opencode/prompts/*.md`, `opencode.json` |

The planner, generator, and healer files in those locations are generated by Playwright and should generally be treated as upstream-generated definitions.

Repository-provided additions and workflow changes should live in the common layer unless the user explicitly asks to customize these tool-specific files.

## Level 3: Generic MCP-capable agents

For Cursor, Windsurf, Cline, Goose, or other MCP-capable agents, adapt the MCP server definition:

```json
{
  "mcpServers": {
    "playwright-test": {
      "command": "npx",
      "args": ["playwright", "run-test-mcp-server"]
    }
  }
}
```

If the agent uses a different schema, preserve the command and args:

```text
command: npx
args: playwright run-test-mcp-server
```

Then attach the role instructions from one of these sources:

- `AGENTS.md`
- `.agents/skills/playwright-service-mapper/SKILL.md`
- `.agents/skills/playwright-test-planner/SKILL.md`
- `.agents/skills/playwright-test-plan-validator/SKILL.md`
- `.agents/skills/playwright-test-generator/SKILL.md`
- `.agents/skills/playwright-test-healer/SKILL.md`

## Required role order for all agents

For whole-service discovery:

```text
service-mapper -> planner -> plan-validator -> generator -> healer
```

For a known feature:

```text
planner -> plan-validator -> generator -> healer
```

Generation is allowed only after validation returns `PASS`.

If validation returns `FAIL`, the next action is not generation. The next action is planner refinement using the validation report.

If validation returns `BLOCKED`, resolve missing context before continuing.

## Agents without MCP support

If an agent cannot connect to MCP, it can still use the role prompts as planning/generation guidance, but it cannot reliably perform browser exploration or Playwright Test MCP generation/healing.

In that case, use it only for:

- reviewing service mapping artifacts
- reviewing test plans
- improving scenario wording
- validating planner output from existing artifacts
- reviewing generated Playwright code
- proposing locator/assertion improvements from provided traces or screenshots

Do not expect MCP-free agents to discover the live UI accurately.

## Recommended maintenance policy

Keep the workflow source of truth simple:

- cross-agent instructions live in `AGENTS.md`
- common reusable workflows live in `.agents/skills/`
- setup and compatibility notes live in `docs/`
- service mapping artifacts live in `artifacts/service-exploration/`
- feature plans live in `specs/`
- validation reports live in `specs/_reviews/`
- upstream Playwright-generated definitions stay in `.claude/`, `.github/`, `.opencode/`

When Playwright updates its official generated files, regenerate those files first, then re-check whether the common compatibility layer still matches the desired service-mapper/planner/validator/generator/healer behavior.
